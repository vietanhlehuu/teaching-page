Tại sao chúng ta cần sử dụng Git?

Nếu bạn đang phát triển dự án từ 2 người trở lên, dùng ngay Git, không thắc mắc, và bạn sẽ tự biết lợi ích của Git là gì.

Với nhu cầu làm việc nhóm nhiều người khi code thì dùng Git là một điều hiển nhiên. Git giúp chúng ta đồng bộ các files và "ghép code tự động".

Với nhu cầu làm việc một mình thì Git vẫn rất quan trọng, nó giúp ta quản lý lịch sử cập nhật các files, để đôi khi chúng ta đã code chạy ngon lành, nhưng việc cập nhật, sửa đổi gây ra lỗi, lúc đó, nhu cầu trở về hiện trạng trước đó là rất cần thiết.

Command line

The command line hay command line interface (CLI) là những thứ chúng ta rất hay bắt gặp trong các phim về chủ đề IT.

Pí: À, ý là mấy cái màn hình mà hacker đồ hay gõ nhìn ngầu ngầu đó.

(2 đoạn dưới này là thông tin thêm, đọc cho biết hoặc bỏ qua cũng được).

--

Các chương trình chạy các lệnh command line đó được gọi là Shell. Còn ứng dụng bọc cái Shell đó được gọi là Terminal (Terminal có thể mở được nhiều Shell, nếu bạn nào đã dùng Visual Studio Code thì có thể đã bắt gặp terminal và mở được nhiều shell trong đó).

Shell có nhiều loại, một số cái thường gặp là Bash, zsh, ...

--

Trong bài này, mình mặc định bạn đã tải Git và bấm Next liên tục 🍌 (nếu bạn chưa biết gì về Git thì chỉ cần làm vậy ✌). Đối với Windows thì việc cài Git sẽ đi kèm ứng dụng Git Bash, nơi mà chúng ta sẽ thực hành, còn đối với những bạn sử dụng hệ điều hành khác thì đã có sẵn các Shell bên trong hệ điều hành rồi.

Bạn tạo một thư mục mới, vào trong đó, nháy phải để chọn Git Bash here. (Còn các bạn dùng hệ điều hành khác thì tự mở terminal và đi vào thư mục đó là được).

Lệnh đầu tiên mình học là cách tạo ra một thư mục mới:

SHELL
mkdir ten-thu-muc

Sau khi nhập và enter lệnh trên, một thư mục tên ten-thu-muc sẽ được tạo.

Để xem danh sách các files và thư mục bên trong thư mục đang đứng, chúng ta gõ:

SHELL
ls

Lệnh ls còn một số option nữa, mà chúng ta sẽ không quan tâm trong bài này, đến lúc biết cơ bản rồi thì bạn có thể tự xem được.

Để tạo một file mới, chúng ta gõ:

SHELL
touch ten-file.txt

Gõ lệnh ls để xem danh sách thư mục và file mới tạo đi nào.

Pí: Ok, chờ chút, nãy giờ hơi bị dài nha, đọc hơi mệt, hơi quạu rồi đó.

Rồi, giờ chúng ta muốn đi vào thư mục ten-thu-muc mà chúng ta đã tạo vừa rồi đó, chúng ta gõ:

SHELL
cd ten-thu-muc

Một tip nhỏ cho các bạn bắt đầu là mình chĩ cần gõ cd te một đoạn rồi nhấn tab nó sẽ tự động gợi ý và điền vào giúp mình.

Pí: Rồi giờ muốn ra lại bên ngoài phải làm sao ba?

Muốn trở lại thư mục trước, chúng ta gõ:

SHELL
cd ..

.. có thể hiểu là đại diện cho thư mục bên ngoài so với vị trí hiện tại, ví dụ giờ chúng ta muốn trở ra thư mục ngoài của thư mục ngoài (ra 2 cấp).

SHELL
cd ..
cd ..

Hoặc

SHELL
cd ../..

Nếu bạn có thực hiện đi ra nhiều quá thì trở lại chỗ thư mục chứa file ten-file để tiếp tục nhé. Giờ chúng ta muốn xóa một file hoặc một thư mục

SHELL
rm ten-file.txt
rm -r ten-thu-muc

Tất nhiên command line còn nhiều lệnh nữa, nhưng nắm cơ bản những lệnh trên là đủ để chúng ta học phần dưới và chém gió với người ngoài.

Sử dụng Vim

Vim là một trình soạn thảo văn bản tích hợp sẵn trong Bash, vì vậy nó khá là phổ biến. Tùy gu mỗi người, nhưng biết những thứ này một chút một ít sẽ tốt hơn. (Nếu bạn cài Git bằng cách bấm Next liên tục thì bắt gặp Vim là một điều khó tránh khỏi - spoil: nhiều người không biết thoát ra như thế nào khi gặp Vim - spoil tiếp: nhấn :q! để thoát).

Sau khi đọc 2 spoil đó, bạn có thể xem phần dưới luôn cũng được, hoặc đọc tiếp phần này để biết một số thao tác cơ bản.

Mở Git Bash, gõ:

SHELL
vim ten-file-gi-do.txt

Sau khi thực hiện lệnh đó, một file tên ten-file-gi-do được tạo ra tạm thời (nếu file đó đã tồn tại trước đó thì nó chỉ mở chứ không tạo ra một file khác nữa) và vim sẽ mở sẵn nó để chúng ta thực hiện các thao tác chỉnh sửa.\

Trong vim có một số chế độ. Để thực hiện chọn một chế độ nào đó, chúng ta nhấn ESC rồi mới chọn chế độ.

Chế độ 1: Chỉnh sửa

Để vào chế độ chỉnh sửa thì chúng ta nhấn i (nếu bạn trước đó có bấm lung tung gì thì nhớ nhấn ESC để trở về trạng thái nhập chế độ nhé).

Sau khi vào chế độ chỉnh sửa, chúng ta có thể dùng các phím mũi tên trái, phải, lên, xuống để di chuyển bên trong văn bản và thêm xóa nội dung.

Chế độ 2: Lưu

Trở lại trạng thái nhập chế độ bằng cách nhấn ESC và nhập :w để lưu (Nhớ nhập dấu : nhé)

Chế độ 3: Thoát

Trở lại trạng thái nhập chế độ bằng cách nhấn ESC và nhập :q để thoát (Nhớ nhập dấu : nhé)

Pí: Vậy tui muốn sau khi chỉnh sửa một file rồi lưu thoát luôn phải làm sao

Muốn lưu và thoát luôn thì chúng ta gõ: :wq

Pí: Nếu tui có chỉnh sửa file rồi mà giờ muốn thoát thôi được không?

Không nhé, nó sẽ yêu cầu thoát phải lưu hoặc thoát không lưu. Lưu rồi thoát mình đã chỉ ở trên. Còn thoát mà không lưu thì chúng ta nhập: :q!

Thôi vậy cũng đủ để dùng Git rồi đó.

Muốn tìm hiểu thêm một số thao tác cơ bản nữa thì có thể đọc ở đây nhé: List of Command Line Commands

Dùng được Git

Pí: Nãy giờ đọc quá trời quá đất giờ mới học nội dung chính đó hả? Quạu rồi nha!

Hãy tạo một thư mục mới ở bất kỳ chỗ nào bạn thích. Vào thư mục vừa tạo, nháy phải chuột và chọn Git Bash here (nếu dùng Windows) hoặc mở Terminal và vào thư mục đó (nếu dùng hệ điều hành khác).

Sử dụng Git đồng nghĩa với việc chúng ta muốn Git quan sát sự thay đổi và chỉ ra sự thay đổi đó cho chúng ta.

Pí: Vậy mình cài Git là nó tự động "theo dõi" cả cái máy mình hay sao?

Không! Để Git có thể quan sát một project, chúng ta cần khởi tạo Git trong thư mục project đó:

SHELL
git init

Giờ tạo một file gì đó mới trong thư mục project của chúng ta nhé. Bạn có thể tạo trực tiếp trên máy bằng giao diện, ở đây mình tạo bằng command line:

SHELL
echo "Absmentor is the second best" > Hello.txt

Lệnh trên đã tạo 1 file mới tên là Hello.txt với nội dung "Absmentor is the second best".

Pí: Vậy làm sao chúng ta kiểm tra xem Git nó đã quan sát những thay đổi gì?

Gõ:

SHELL
git status

Chúng ta sẽ được hiển thị thông báo dạng như thế này

TEXT
On branch master
No commits yet
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        Hello.txt
nothing added to commit but untracked files present (use "git add" to track)
Pí: Nó biết vừa có một file "Hello.txt" được thêm kìa, ghơ ghơ

Giờ chúng ta muốn nó lưu lại thay đổi đó, coi như đó là "chốt đơn" phiên bản 1, để chúng ta có sửa gì tiếp thì chúng ta có thể so sánh với phiên bản đó chứ không phải so sánh từ lúc khởi tạo ban đầu.

Pí: Câu trên nghe hơi khó hiểu đó nghen, thôi kệ, để tui đọc thêm lần nữa coi hiểu không.

Nói chung là mình nhờ nó lưu lại thôi đó mà.

Giờ muốn Git lưu, nó cần 2 thông tin:

  1. File nào cần lưu
  2. Lưu với thông tin gì (Để lần sau mày tìm lại thì dễ cho mày thôi - Git nói)

Bước 1: Bảo Git mình cần lưu những file nào:

SHELL
git add .

Khi thực hiện câu lệnh trên, Git sẽ xem những file nào từ thư mục đó trở vào các thư mục con đang bị thay đổi và ghi nhận sự thay đổi đó.

Nếu chúng ta chỉ muốn Git ghi nhận 1 file cụ thể thì:

SHELL
git add Hello.txt

Việc này giống như bạn muốn Gmail lưu thư nháp giùm cho bạn vậy, bạn sẽ thêm các file đính kèm là các file đã thay đổi. Tiếp theo, bạn cần cho nó một chủ đề, mô tả.

Git cũng cần mô tả đó:

SHELL
git commit -m "Mô tả gì đó để dễ nhớ"

(m trong -m viết tắt cho message: thông điệp, mô tả)

Bước này mình mình hay gọi là commit.

Tuy nhiên, nếu bạn lần đầu cài Git, và bạn chưa cấu hình gì cho nó cả, đồng nghĩa với việc nó chẳng biết bạn là ai. Như bạn muốn Gmail lưu giùm bạn nhưng bạn chưa đăng nhập vậy. Lúc này bạn phải cho Git biết bố mày, à nhầm, bạn là ai.

SHELL
git config --global user.name "Ten cua ban"
git config --global user.email "Email cua ban"

(Thao tác này chỉ để Git ghi nhận thông tin người tạo ra commit đó là ai thôi, chứ nó không yêu cầu phải xác thực đúng là tên đó hay email đó)

Sau khi thêm 2 bước đó, cũng như Gmail, nó sẽ lưu tự động lại thư nháp đó cho bạn. Bởi vậy luôn nhớ, đầu tiên phải add những file thay đổi, thứ hai commit một thông điệp.

Giờ chúng ta kiểm tra lại xem còn có gì thay đổi so với lần cuối chúng ta lưu vào Git hay không:

SHELL
git status

Và kết quả là không có gì thay đổi:

TEXT
On branch master
nothing to commit, working tree clean

Chúng ta có thể xem lại lịch sử những lần chúng ta yêu cầu Git lưu:

SHELL
git log
TEXT
Author: Lê Hữu Việt Anh <lehuuvietanh@gmail.com>
Date:   Sat Jun 20 13:43:04 2020 +0700
    Mô tả gì đó cho dễ nhớ

Nếu chúng ta commit lần 2, lần 3, nó sẽ hiển thị khi log ra như trên. Mỗi commit đó mang một trạng thái tại thời điểm bạn yêu cầu Git lưu, vì vậy, đôi khi gặp một vấn đề nào đó, bạn có thể trở lại trạng thái trước đó bằng cách nhảy về commit bạn muốn. (Việc nhảy về commit trước là một câu lệnh nâng cao, với nhu cầu cơ bản bạn chưa cần dùng nó - và câu lệnh đó mình sẽ viết ở phần dưới).

Khúc tiếp theo là một phần khi đi làm thực tế các bạn sẽ gặp.

Nhánh trong Git

Hãy tưởng tượng như thế này (nếu bạn chưa biết nhánh hoặc chưa từng tạo nhánh trong Git): Bạn đã phát triển xong 1 dự án, và dự án đó được tung ra cho khách hàng dùng. Ví dụ điển hình là Facebook. Tuy nhiên, sau đó bạn muốn phát triển thêm tính năng và giao diện mới. Nhưng tính năng và giao diện mới bạn muốn nó khác hoàn toàn và sẽ tung ra trong 1 năm nữa.

Trong quá trình bạn phát triển tính năng mới đó, trang web bạn đã tung ra (release) dùng trước đó đang bị lỗi. Bạn đành phải sửa lỗi và triển khai (deploy) lại để người dùng có thể tiếp tục dùng ngon lành hơn. Vấn đề ở đây là dự án hiện tại của bạn đang có một đống code mới (có thể có rất nhiều lỗi khác trong code mới đó), nếu giờ bạn triển khai lỗi bạn vừa sửa cho trang hiện tại thì đồng nghĩa bạn sẽ triển khai luôn đống tính năng bạn đang làm dang dở.

Để tránh tình trạng đó, chúng ta có thể tách những file trong project chúng ta thành 2 hoặc nhiều luồng khác nhau, chúng ta có thể thêm, xóa, sửa những file trong luồng này mà không ảnh hưởng đến luồng kia. Ví dự án như con đường vậy, một thời điểm nào đó mình muốn nó tách thành 2 đường khác để dùng với những mục đích khác nhau, đến lúc nào đó lại muốn nó hợp lại với nhau.

Thì chuyện tách thành các luồng độc lập hay các đường độc lập như vậy gọi là tách nhánh branch. Còn sau khi phát triển 2 nhánh một thời gian và muốn hợp lại thì gọi là merge.

Mặc định khi khởi tạo Git cho thư mục dự án, một nhánh sẽ được sinh ra và chúng ta làm trên nhánh đó: master. Kiểm tra mình đang ở nhánh nào:

SHELL
git branch

Kết quả:

TEXT
* master

Giờ muốn tạo nhánh mới và nhảy qua làm việc ở nhánh đó luôn:

SHELL
git checkout -b feature

Mình đặt tên cho nhánh mới đó là feature. Kiểm tra lại nhánh mình đang đứng:

SHELL
git branch

Kết quả:

TEXT
* feature
  master

Giờ những gì chúng ta cập nhật ở đây sẽ không ảnh hưởng đến nhánh master.

Bạn có thể thêm một file khác trong nhánh này:

SHELL
touch feature.txt
git add feature.txt
git commit -m "Add feature"
git log

Kết quả:

TEXT
commit e8d8e3a4620accd140554134c913787697a28f22 (HEAD -> feature)
Author: Lê Hữu Việt Anh <lehuuvietanh@gmail.com>
Date:   Sat Jun 20 14:14:42 2020 +0700
    Add feature
commit 7c512ec2f34719406eecb4bbe0e488f14753a098 (master)
Author: Lê Hữu Việt Anh <lehuuvietanh@gmail.com>
Date:   Sat Jun 20 13:43:04 2020 +0700
    Mô tả gì đó cho dễ nhớ

Giờ chúng ta muốn quay lại nhánh master và kiểm tra lịch sử commit bên nhánh này:

SHELL
git checkout master
git log

Kết quả:

TEXT
Author: Lê Hữu Việt Anh <lehuuvietanh@gmail.com>
Date:   Sat Jun 20 13:43:04 2020 +0700
    Mô tả gì đó cho dễ nhớ

Vậy là nhánh master không bị ảnh hưởng bởi nhánh feature kia. Chúng ta thêm một file trong nhánh master:

SHELL
echo "Haha, day la mot file moi de master co them mot commit" > another-file.txt
git add .
git commit -m "Add a new file on master"
git log

Và kết quả:

TEXT
commit b501a4d111e587c542a94f5d5935515d592e91c8 (HEAD -> master)
Author: Lê Hữu Việt Anh <lehuuvietanh@gmail.com>
Date:   Sat Jun 20 14:18:48 2020 +0700
    Add a new file on master
commit 7c512ec2f34719406eecb4bbe0e488f14753a098
Author: Lê Hữu Việt Anh <lehuuvietanh@gmail.com>
Date:   Sat Jun 20 13:43:04 2020 +0700
    Mô tả gì đó cho dễ nhớ

Vậy là giờ chúng ta có 2 nhánh: master và feature.

2 nhánh đó chứa các commit như sau:

  1. master: Mô tả gì đó cho dễ nhớ --> Add a new file on master
  2. feature: Mô tả gì đó cho dễ nhớ --> Add feature

Giờ chúng ta đang đứng ở nhánh master và có nhu cầu cập nhật commit từ nhánh feature, chúng ta phải merge nhánh feature vào master:

SHELL
git merge feature

Sau khi merge thì nó sẽ tự động tạo một commit mới, và nó cho phép mình sửa bằng cách tự động mở Vim (hoặc trình soạn thảo văn bản nào đó nếu bạn có cài đặt). Nếu nó mở Vim, bạn thoát ra bằng cách :q!.

Phần này thắc mắc hay gặp lỗi bạn có thể comment phía dưới nhé.

Đó là sử dụng Git ở dưới máy (local). Nếu bạn làm nhóm, bạn sẽ phải dùng thêm một trang lưu trữ online dành cho Git nữa, nó có thể là: Github, Gitlab, Bitbucket, ...
